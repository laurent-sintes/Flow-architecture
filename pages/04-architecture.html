4. Architecture cible

Une architecture orientée engagement, règles, événements et intégration progressive avec le cœur ERP.

4.1. Vue d’ensemble

FLOW se positionne entre les canaux de vente, les systèmes cœur et les acteurs d’exécution. Il ne remplace pas l’ERP : il porte la logique d’engagement, d’orchestration et de suivi transverse.

ERP horizontal transactionnel + FLOW vertical d’engagement et d’orchestration.

4.2. Blocs logiques

  • Experience APIs : interfaces exposées aux canaux et frontaux.
  • Case Management : cycle de vie de la demande, décisions, exceptions, statuts métier.
  • Rules & Decisioning : règles de promesse, allocation, priorisation, éligibilité.
  • Inventory Visibility : stock consolidé, stock réservé, stock promis, projection.
  • Event Bus : publication des événements métier et intégration asynchrone.
  • MDM / référentiels : produit, client, point de stockage, organisation, contrat.
  • Data Hub : observabilité métier, performance, apprentissage, pilotage.

4.3. Patterns d’intégration

  • API synchrone pour la promesse et les décisions nécessitant une réponse immédiate.
  • Événements métier pour propager les changements d’état significatifs.
  • CRE pour la traduction finance/gestion des événements opérationnels.
  • Anti-corruption layer entre FLOW et les ERP/WMS/OMS existants.
  • Observabilité transverse : corrélation case, commande, stock, expédition, finance.

4.4. Candidats techniques

À discuter selon les standards groupe : Spring Boot / Quarkus, Kafka, Drools/DMN, Temporal si besoin d’orchestration technique durable, API Gateway, OpenAPI/AsyncAPI, data lakehouse et observabilité applicative.
